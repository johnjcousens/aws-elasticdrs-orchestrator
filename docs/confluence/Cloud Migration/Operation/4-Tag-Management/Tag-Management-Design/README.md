| Tags are the building blocks of cloud resource reporting, but they are also functional and used in everything from auto-scaling to license management. Defining a tagging schema for your organization that overlays your business and financial attributes to your cloud usage is a Well-Architected best practice. Due to the ephemeral nature of resources in the cloud, it is not necessary to log cloud resources in the same static pattern as physical resources. Instead, a tagging strategy is employed to log resources for tracking, dependency, and chargeback purposes. Additionally, the attributes of the cloud resources being utilized can be readily identified using AWS Config, alleviating the need for manual entry and low-value data entry time.  Customers that are most effective in their use of tags typically create business-relevant tag groupings to organize their resources along technical, business, and security dimensions. Also, customers that use automated processes to manage their infrastructure also include additional, automation-specific tags to aid in their automation efforts.  Key Elements to a Cloud Governance Framework include at least below types:   * Technical * Automation * Business * Security   Tags are used by many groups including; Finance, Operations, IRM/Security, Application Owners, and others. The following table provides details on each tag type. |

Optional Tags
-------------

All tags that listed in “**HealthEdge Tag Types**” in above section are considered as optional tags, each BUs/Teams should use with listed tag names and values to make cross-organization consistency.

Any tags that not listed “**HealthEdge Tag Types**”, please submit request ticket to get approved and added to the “**HealthEdge Tag Types**” before put in use.

Tagging Your Resources
----------------------

Nearly every AWS resource supports tagging.  You should refer to the AWS documentation for AWS resources you plan to use for specific information.  Some AWS resources include system created tags.  For example, EC2 instances include the **aws:createdBy** tag.  [Refer to the documentation](https://docs.aws.amazon.com/general/latest/gr/aws_tagging.html#tag-conventions) for more details.

You should tag your resources when they are provisioned using your tagging standard.  You can enforce your tagging standards in multiple ways.  For example, you can use CloudFormation templates to provision your resources and specify your required tags in your template.  You can allow users to enter or select values for the tag values in your template.  Similarly, you can use AWS Service Catalog to provision AWS resources with TagOptions that align to your tagging standard.

If you need to tag multiple resources in AWS, you can use the [AWS Tag Editor](https://docs.aws.amazon.com/ARG/latest/userguide/tag-editor.html).  This can be especially helpful if you need to remediate untagged resources to align them to your tagging standard.

Tagging Methods
===============

AWS users creating new AWS resources will be responsible for following the tagging standards using any available provisioning method.

In HealthEdge, All AWS resources in AWS accounts under “Workloads“ OU and its child OUs should be provisioned via CDK. In CDK workflow, a cross account role of “`arn:${PARTITION}:iam::*:role/CrossAccountDeploymentRole`“ will be assumed by CDK pipelines for resources provisioning, all the mandatory tags(according to specific OU) should have proper tagging to avoid providing failure.

**NOTE: Manual provisioning in AWS Console is generally not allowed, except for accounts under Sandbox OU.**

Tag Governance
==============

Tag compliance to tagging standards have been implemented at the AWS Organization level using Tag policies. The tagging standards and tag policies apply to the entire Workloads OU and Sandbox OU. Along with AWS Security Control Policy (SCP), if resources is missing mandatory tags, will be denied for creation, the control is applied to supported resources like EC2, EKS, Lambda, etc.

| **Recommendations for Future Enhancements** |
| --- |
| An initial project, post Operational Readiness State, is to review all instances deployed across all of the \[Customer\] AWS accounts for appropriate tags. Using AWS Config, instances identified as not adhering to the established tagging standards should be traced to the owner and either updated or decommissioned. Following an initial review, grace and education period, AWS Lambda and AWS Config can be deployed to automatically delete instances which do not use proper tags.    Tagging is not a static activity. The Cloud Platform Team, working with the corresponding security and asset management teams should periodically review the list of tags to determine the appropriateness of the tags. |
